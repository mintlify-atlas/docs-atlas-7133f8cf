Sortvis is engineered for high-performance visualization of sorting algorithms, capable of smoothly animating hundreds of elements even at the fastest speeds. This guide explores the key optimizations and architectural decisions.

## Uint8Array for efficient storage

Sortvis uses `Uint8Array` typed arrays instead of regular JavaScript arrays for storing both array values and color states:

```typescript src/lib/store/visualizer.svelte.ts:48-62
export class VisualizerState {
  /**
   * `array` and `colors` are plain (non-reactive) `Uint8Array` fields.
   * `Uint8Array` is used because bar values (5–104) and color states (0–5) are
   * single-byte integers — 8x smaller than `number[]` (float64) and pointer arrays,
   * making snapshot copies in the canvas a native `memcpy`. Elements are mutated
   * in-place during each sort step; the `tick` signal below notifies the canvas
   * when to re-render. Deep reactivity and `$state.raw` are both avoided to prevent
   * per-mutation reactivity overhead in the high-frequency sort engine loop.
   */
  array: Uint8Array = new Uint8Array(0);
  initialArray: Uint8Array = new Uint8Array(0);
  colors: Uint8Array = new Uint8Array(0);
  tick = $state(0);
}
```

### Color state encoding

Color states are encoded as integers (0-5) instead of strings:

```typescript src/lib/types/visualizer.ts:22-29

```

This allows storing colors in a `Uint8Array` alongside the array values, enabling fast lookups:

## Batch processing at different speeds

Sortvis processes multiple sort steps per frame at higher speeds to overcome the `requestAnimationFrame` limit (~60 FPS):

```typescript src/lib/store/visualizer.svelte.ts:239-251
private runLoop() {
  if (this.#status !== 'playing') return;

  // Grab the current speed config based on the discrete setting
  const { delay, batch } = SPEED_CONFIG[this.#speed];
  for (let i = 0; i < batch && this.step(); i++);

  if (this.#status === 'playing') {
    // Sync internal counters to reactive $state once per batch (not once per step)
    this.stats = { comparisons: this._comparisons, swaps: this._swaps };
    this.tick++;
    this.loopTimeoutId = setTimeout(() => this.runLoop(), delay);
  }
}
```

### Speed configurations

<AccordionGroup>
  <Accordion title="Slow speeds (snail, slow, med)">
    Process 1 step per batch with delays between frames. This provides smooth, visible animations where each operation can be clearly observed.
    
    ```typescript
    snail: { delay: 500, batch: 1 },
    slow: { delay: 100, batch: 1 },
    med: { delay: 10, batch: 1 },
    ```
  </Accordion>
  
  <Accordion title="Fast speed">
    Process 1 step per frame with no delay. Limited only by `requestAnimationFrame` (typically 60 FPS), resulting in 60 steps per second.
    
    ```typescript

    ```
  </Accordion>
  
  <Accordion title="Rapid speed">
    Process 5 steps per frame with no delay. Achieves ~300 steps per second (5 × 60 FPS), making sorts complete much faster.
    
    ```typescript

    ```
  </Accordion>
  
  <Accordion title="Flash speed">
    Process 100 steps per frame with no delay. Achieves ~6,000 steps per second (100 × 60 FPS). Most sorts complete in seconds even for large arrays.

  </Accordion>
</AccordionGroup>

<Info>
  Statistics are synchronized to reactive state only once per batch, not once per step. This prevents excessive reactivity overhead during high-frequency updates at rapid and flash speeds.
</Info>

## Visualization engine architecture

The rendering system uses a "sleepy loop" pattern combined with intelligent diffing to minimize CPU usage and maximize performance.

### Sleepy loop pattern

The renderer only runs when necessary:

```typescript src/lib/components/layout/canvas/renderer.svelte.ts:77-96
function loop() {
  if (dirty) {
    dirty = false;
    drawFrame();
  }

  // Keep loop running if playing (animation updates happen frequently)
  // OR if dirty flag is set (pending update)
  if (visualizer.status === 'playing' || dirty) {
    rafId = requestAnimationFrame(loop);
  } else {
    // Stop the loop — truly 0 CPU usage when idle
    rafId = null;
  }
}

function startLoop() {
  if (rafId === null) loop();
}
```

**Benefits:**

- **Zero CPU when idle**: The loop stops completely when the animation is paused or finished.
- **Automatic wake-up**: The loop restarts when the visualization becomes dirty (tick changes) or starts playing.
- **Battery-friendly**: Mobile devices benefit significantly from the reduced CPU usage.

### Intelligent diffing

The renderer compares the current state with the previously drawn state to minimize canvas operations:

```typescript src/lib/components/layout/canvas/renderer.svelte.ts:38-75
function drawFrame() {
  if (!ctx) return;

  const width = canvas.width;
  const height = canvas.height;
  const { array, colors } = visualizer;

  if (array.length === 0) return;

  if (fullRedraw || !cachedLayout || prevColors.length !== colors.length) {
    // Full redraw — resize, theme change, array regeneration, first draw
    ctx.clearRect(0, 0, width, height);
    cachedLayout = drawAllBars(ctx, width, height, array, colors, themeColors);
    fullRedraw = false;
  } else {
    // Diff against what was last drawn to find changed bars
    const changed: number[] = [];
    for (let i = 0; i < array.length; i++) {
      if (colors[i] !== prevColors[i] || array[i] !== prevArray[i]) changed.push(i);
    }

    if (changed.length === 0) return; // Nothing changed

    if (changed.length > array.length * 0.4) {
      // More than 40% changed — full redraw is faster (batched by color)
      ctx.clearRect(0, 0, width, height);
      cachedLayout = drawAllBars(ctx, width, height, array, colors, themeColors);
    } else {
      // Partial redraw — only the changed bars
      drawBarsByIndex(ctx, changed, cachedLayout, array, colors, themeColors);
    }
  }

  // Snapshot what we just drew
  prevColors = colors.slice();
  prevArray = array.slice();
}
```

<Steps>
  <Step title="Full redraw triggers">
    Full redraws occur on resize, theme change, array size change, or first render. This ensures the canvas is always correctly initialized.
  </Step>
  
  <Step title="Diff detection">
    The renderer compares current `array` and `colors` with `prevArray` and `prevColors` to find changed bars.
  </Step>
  
  <Step title="Smart rendering strategy">
    If fewer than 40% of bars changed, only those bars are redrawn. Otherwise, a full redraw is faster.
  </Step>
  
  <Step title="Snapshot caching">
    After drawing, the current state is copied to `prevArray` and `prevColors` using native `Uint8Array.slice()` for fast memcpy.
  </Step>
</Steps>

### Partial vs. full redraw

The 40% threshold is a performance optimization:

- **Partial redraw**: Clears and redraws only changed bars individually. Efficient when few bars change.
- **Full redraw**: Clears the entire canvas and redraws all bars batched by color. More efficient when many bars change because batching reduces draw calls.

```typescript src/lib/components/layout/canvas/draw.ts:33-70
export function drawAllBars(
  ctx: CanvasRenderingContext2D,
  width: number,
  height: number,
  array: Uint8Array,
  colors: Uint8Array,
  themeColors: Record<BarState, string>
): BarLayout {
  const layout = computeLayout(width, height, array);
  const size = array.length;

  const batches: Record<string, Path2D> = {};

  for (let i = 0; i < size; i++) {
    const value = array[i]!;
    const state = BAR_STATES[colors[i]!] ?? 'default';
    const color = themeColors[state];

    batches[color] ??= new Path2D();

    const barHeight = ((value / layout.maxValue) * (height - CONFIG.MIN_BAR_HEIGHT_OFFSET)) | 0;
    const x0 = (i * layout.step) | 0;
    batches[color].rect(
      x0,
      height - barHeight,
      (((i + 1) * layout.step) | 0) - x0 - layout.gap,
      barHeight
    );
  }

  // Draw all bars of each color in a single fill() call
  for (const [color, path] of Object.entries(batches)) {
    ctx.fillStyle = color;
    ctx.fill(path);
  }

  return layout;
}
```

<Info>
  Color batching reduces the number of `fillStyle` assignments and `fill()` calls. For example, a 512-element array with 6 colors requires only 6 draw calls instead of 512.
</Info>

## Additional optimizations

### Non-reactive arrays with tick signal

The `array` and `colors` fields are plain (non-reactive) `Uint8Array` fields. A separate `tick` counter is used to signal when the renderer should update:

```typescript src/lib/store/visualizer.svelte.ts:59-62
array: Uint8Array = new Uint8Array(0);
initialArray: Uint8Array = new Uint8Array(0);
colors: Uint8Array = new Uint8Array(0);
tick = $state(0);
```

This avoids the overhead of deep reactivity on every array mutation. The renderer reactively watches `tick`:

```typescript src/lib/components/layout/canvas/renderer.svelte.ts:137-142
$effect(() => {
  void visualizer.tick;
  dirty = true;
  // Don't set fullRedraw — let drawFrame diff and decide
  startLoop();
});
```

<Warning>
  Deep reactivity (tracking every array element mutation) would cause severe performance degradation during high-frequency updates. The tick signal pattern is essential for maintaining 60 FPS even at flash speed.
</Warning>

### Integer-only math

All calculations use integer arithmetic with bitwise OR truncation:

```typescript src/lib/components/layout/canvas/draw.ts:54-61
const barHeight = ((value / layout.maxValue) * (height - CONFIG.MIN_BAR_HEIGHT_OFFSET)) | 0;
const x0 = (i * layout.step) | 0;
batches[color].rect(
  x0,
  height - barHeight,
  (((i + 1) * layout.step) | 0) - x0 - layout.gap,
  barHeight
);
```

The `| 0` operator truncates to a 32-bit integer, which is faster than `Math.floor()` and produces pixel-aligned values for crisp rendering.

### DPR capping

Device pixel ratio is capped at 2× to prevent excessive memory usage on high-DPI displays:

```typescript src/lib/components/layout/canvas/renderer.svelte.ts:13
const dpr = Math.min(window.devicePixelRatio || 1, 2);
```

This balances visual quality with performance. Most users won't notice the difference between 2× and 3× rendering, but memory usage and draw time increase quadratically with DPR.

### ResizeObserver with devicePixelContentBoxSize

The renderer uses the modern `devicePixelContentBoxSize` API when available for precise physical pixel dimensions:

```typescript src/lib/components/layout/canvas/renderer.svelte.ts:103-116
const resizeObserver = new ResizeObserver((entries) => {
  const entry = entries[0];
  if (!entry) return;

  // Prefer devicePixelContentBoxSize (exact physical pixels) when available,
  // falling back to contentRect * dpr for older browsers
  const dpBox = entry.devicePixelContentBoxSize?.[0];
  canvas.width = dpBox ? dpBox.inlineSize : entry.contentRect.width * dpr;
  canvas.height = dpBox ? dpBox.blockSize : entry.contentRect.height * dpr;

  // Draw immediately to avoid blank flash — don't wait for next RAF
  fullRedraw = true;
  drawFrame();
});
```

This eliminates rounding errors and ensures pixel-perfect rendering on all devices.

## Performance characteristics

With these optimizations, Sortvis achieves:

- **60 FPS** at slow to fast speeds on arrays up to 512 elements
- **~6,000 steps/second** at flash speed (100 steps × 60 FPS)
- **0% CPU usage** when idle or paused
- **Less than 1MB memory** even for the largest arrays
- **Instant response** to user interactions (play, pause, speed changes)

<Note>
  The performance optimizations make it possible to visualize sorting algorithms in real-time without compromising the educational value or user experience.
</Note>
